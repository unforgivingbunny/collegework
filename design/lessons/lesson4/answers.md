# Самостійна робота студента

## 1. Контрольні запитання

- **Чим відрізняється User Flow від Task Flow?**
  
  *User Flow* описує повний шлях користувача через весь продукт, включаючи різні сценарії, емоції та розгалуження (наприклад, шлях від реєстрації до покупки). *Task Flow* — це спрощена, лінійна діаграма виконання одного конкретного завдання без відгалужень (наприклад, схема "скидання пароля").

- **У яких випадках доцільно використовувати Low-Fi?**
  
  Low-Fi (низької деталізації) доцільно використовувати на ранніх етапах проєктування: для швидкого генерування ідей, перевірки базової структури, командної роботи ("дизайн-спринти"), або коли важливий зворотний зв'язок щодо концепції, а не візуального стилю.

- **Чому використання тексту "Lorem Ipsum" може зашкодити при створенні вайрфреймів?**
  
  Lorem Ipsum ігнорує семантику та довжину реального контенту. Це може призвести до неправильного оцінювання ієрархії тексту, переносів, скролу та розмірів блоків. Реальний контент часто буває довшим або містить ключові слова, які впливають на UI (наприклад, дати, імена, цифри).

- **Що таке Wireflow?**
  
  Wireflow — це гібрид вайрфрейму (схематичного зображення інтерфейсу) та флоу (схеми переходів). Він показує не окремі екрани, а зв'язки між ними із зазначенням дій користувача (кліки, свайпи), що дозволяє оцінити навігацію без малювання піксельно точного UI.

- **Чому паперовий та інтерактивний прототип економить бюджет?**
  
  Паперовий прототип коштує копійки та дозволяє знайти помилки до початку верстки/кодування. Інтерактивний прототип дозволяє провести юзабіліті-тестування без написання коду. Виправлення помилки на етапі прототипу в сотні разів дешевше, ніж переробка готового ПЗ.

## 2. Питання на дослідження

- **У яких випадках паперові прототипи ефективніші за цифрові?**
  
  Паперові прототипи ефективніші, коли:
  - Потрібна максимальна швидкість (5 хвилин на ітерацію).
  - Ви працюєте з фокус-групою, яка не звикла до софту (або навпаки — з маленькими дітьми).
  - У команді є люди без технічних навичок (маркетологи, юристи, замовник).
  - Ви тестуєте не інтерфейс, а логіку сценарію ("зробити замовлення через хаос").
  - У вас обмежений бюджет або немає доступу до цифрових інструментів.

- **Дослідіть сервіс Balsamiq Wireframes: чим він відрізняється від Figma і чому його досі використовують аналітики?**
  
  *Відмінності Balsamiq від Figma:*
  - Balsamiq спеціально малює "неохайні" вайрфрейми (ефект начерку олівцем), що сигналізує: "Це не фінальний дизайн, критикуйте суть".
  - Figma — потужний векторний редактор для UI/UX, Balsamiq — вужчий інструмент лише для вайрфреймів.
  - Balsamiq працює офлайн, легший, не відволікає на налаштування шрифтів/тіней.
  
  *Чому використовують аналітики:*
  - Аналітикам не потрібен візуал, їм важлива логіка та бізнес-процеси.
  - Balsamiq має готові бібліотеки UI-патернів (алерти, таблиці, чарти) і простий експорт у специфікації.
  - Він інтегрується з Confluence/Jira, що зручно для BA/аналітиків у великих компаніях.

- **Що таке "Двері Нормана" (Norman Doors) у контексті UX-флоу?**
  
  "Двері Нормана" — термін, запроваджений Доном Норманом (книга "Дизайн звичних речей"). Це двері, дизайн яких не підказує, треба їх штовхати чи тягнути (або вони підказують неправильну дію). У контексті UX-флоу це метафора *поганого аффордансу*: коли інтерфейс (кнопка, посилання) виглядає клікабельним, але не є таким, або вимагає неочевидної дії. Наприклад, текст із підкресленням як синій гіперпосилання, але без href. Такі помилки ламають User Flow, викликаючи фрустрацію.

- **Як прототипування впливає на зменшення ризиків у SDLC?**
  
  Прототипування зменшує ризики в SDLC (Software Development Life Cycle) наступним чином:
  - **Ризик неправильних вимог:** прототип дозволяє замовнику "помацати" майбутній продукт до стадії розробки.
  - **Технічний ризик:** виявлення складних інтеграцій на прототипі (наприклад, "анімація тут неможлива через бекенд").
  - **Ризик бюджету та строків:** зміна прототипу коштує годин, зміна коду — днів/тижнів.
  - **Ризик юзабіліті:** раннє тестування на прототипі відсіває 80% сценаріїв із помилками навігації.

- **Знайдіть у мережі приклад "Monster User Flow" великого сервісу (наприклад, Uber чи Spotify). Проаналізуйте, як дизайнери групують блоки, щоб схема залишалася читабельною.**
  
  *Приклад:* Monster User Flow від **Spotify** (схема онбордингу + створення плейлиста + пошук + підписка).

  *Методи групування для читабельності:*
  - **Вертикальні "доріжки" (Swimlanes):** окремі зони для "гостя", "користувача Free", "користувача Premium".
  - **Кольорове кодування:** один колір — один сценарій (жовтий = помилка, зелений = успіх).
  - **Модульність:** великий флоу розбивають на підфлоу (наприклад, "Авторизація", "Пошук треку"), які малюють окремими блоками з посиланнями на наступний аркуш.
  - **Фрейми та контейнери:** повторювані дії (наприклад, "підтвердити клік") виносять в легенду як макрос.
  - **Мінімізація ліній:** замість хаосу ліній використовують нумерацію кроків або явні точки входу/виходу.
